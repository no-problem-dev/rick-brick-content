## はじめに：「AIにコードを書いてもらう」の現実

「AIにプロンプトを投げればコードが書ける時代が来た」——そう興奮したのはいつ頃だろうか。

実際には2025〜2026年にかけて、多くの開発チームが厳しい現実に直面している。Y Combinatorの2025年冬コホートのうち25%が「コードベースの95%をAI生成」で立ち上げたにもかかわらず、それらのチームは技術的負債、セキュリティホール、そして「コンパイルは通るが問題を解いていない実装」に溺れていた。

PVS-Studioの調査では、Copilotを使ったプロジェクトで**バグが41%増加**した事例が報告され、別の研究では**生産性が19%低下**したという結果も出ている。

何が問題だったのか。答えはシンプルだ。**仕様書がなかった**。

本記事では、AIとの協働においてなぜ仕様書が鍵になるのか、そして「Spec-Driven Development（SDD）」という新しいパラダイムの全貌を解説する。

---

## Vibe Codingとその限界

まず「Vibe Coding」を定義しよう。これは、**自然言語で直感的にAIに指示し、生成されたコードをほぼそのまま使う開発スタイル**だ。

プロトタイプや個人プロジェクトでは威力を発揮する。しかし本番システムにおいてはいくつかの根本的な問題が顕在化する。

### 問題1：コンテキストの喪失

AIとの会話はセッションごとにコンテキストがリセットされる（あるいは非常に長い会話では劣化する）。前回「認証はJWTで実装する」と決めたことを次のセッションのAIは知らない。結果として、前回の設計判断と矛盾する実装が生まれ続ける。

### 問題2：合理的な誤った仮定

AIは指示が曖昧な場合、「合理的な仮定」をして実装する。これは開発者の意図と食い違うことがあるが、コードは動作するため気づきにくい。Stack Overflow Blogが指摘した「**製品は作れるがデバッグできない**」という状況が生まれる。

### 問題3：アーキテクチャの一貫性欠如

複数回の会話や複数のAIエージェントが関わる場合、全体のアーキテクチャ設計が散らばる。ある部分はRESTで、別の部分はGraphQLで、また別の部分はgRPCで——という統一性のないシステムができあがる。

### 問題4：「幻覚ループ」

マルチエージェント環境での特有の問題として、**エージェント同士が互いの間違いに同意してしまう「幻覚ループ」**がある。一方のエージェントが誤った実装をし、もう一方がそれをレビューしてOKを出す——という悪循環だ。

```mermaid
graph LR
    A[Agent A: 誤った実装] --> B[Agent B: レビュー]
    B --> C[「問題なし」と承認]
    C --> D[誤実装が本番へ]
    D --> E[バグ発見時には手遅れ]
```

## Spec-Driven Developmentとは何か

SDDはこれらの問題に対する根本的な答えだ。

**Spec-Driven Development（仕様駆動開発）** は、仕様書をソースオブトゥルースとして扱い、コードをその副産物として位置づける開発パラダイムだ。Thoughtworksはこれを「**2025年のAI支援開発における最重要な新プラクティス**」と位置づけている。

核心的な考え方はシンプルだ。

> 「LLMはパターン補完には優れるが、心を読むことはできない」— GitHub

AIはコンテキストが与えられれば非常に有能だが、そのコンテキストを自分で推測することはできない（あるいは推測した結果が意図と異なる）。だからこそ、人間が仕様書という形でコンテキストを明示的に与える必要がある。

### Vibe Coding vs SDD の比較

| 側面 | Vibe Coding | Spec-Driven Development |
|------|-------------|------------------------|
| 情報の主な担体 | 会話・プロンプト | 仕様書ファイル |
| コンテキストの持続性 | セッション内のみ | 永続（ファイルとして保存） |
| 設計判断の記録 | なし（暗黙） | 明示的に文書化 |
| AIへの指示 | 都度プロンプト | 仕様書を参照させる |
| レビューの対象 | コード | 仕様書（先） → コード（後） |
| スケール | 個人・小規模 | チーム・本番システム |

## 仕様書の6つの必須要素

GitHubが2,500以上のリポジトリを分析して特定した、AIが最も有効に活用できる仕様書の構成要素がある。これは「AGENTS.md」「CLAUDE.md」「copilot-instructions.md」といった、AIコーディングアシスタント向けの永続的コンテキストファイルの分析から導き出されたものだ。

### 1. コマンド (Commands)

ビルド、テスト、リント等の実行可能なコマンドを明示する。

```markdown
## コマンド
- ビルド: `npm run build`
- テスト（単体）: `npm test`
- テスト（E2E）: `npm run test:e2e`
- リント: `npm run lint`
- 型チェック: `npm run typecheck`
```

AIにとって「どうやって動作確認すれば良いか」が明確になり、実装後の自己検証が可能になる。

### 2. テスト (Tests)

テストの書き方と期待される出力を定義する。

```markdown
## テスト規約
- フレームワーク: Vitest
- 配置: src/**/*.test.ts
- カバレッジ目標: 80%以上
- モック: vi.mock() を使用、外部APIは必ずモック
```

### 3. プロジェクト構造 (Project Structure)

ディレクトリ構成と各ファイルの役割を記述する。

```markdown
## ディレクトリ構成
src/
├── components/  # UIコンポーネント（ロジックなし）
├── pages/       # ページルーティング
├── utils/       # ユーティリティ（副作用なし）
└── services/    # APIクライアント・ビジネスロジック
```

### 4. コードスタイル (Code Style)

フォーマット規約、命名規則、禁止パターンを明記する。

```markdown
## コードスタイル
- インデント: 2スペース
- 命名: camelCase（変数・関数）、PascalCase（クラス・型）
- `any` 型の使用禁止
- コメント: JSDoc形式（公開API必須）
```

### 5. Gitワークフロー (Git Workflow)

ブランチ戦略、コミットメッセージ規約を定義する。

```markdown
## Gitワークフロー
- ブランチ: feat/*, fix/*, chore/*（develop から切る）
- コミット: conventional commits（feat: / fix: / chore: など）
- PR: develop ベース、squash merge
```

### 6. 境界線 (Boundaries)

最も重要とも言える要素。AIが「常に実行して良いこと」「事前確認が必要なこと」「絶対に行ってはいけないこと」を3段階で定義する。

```markdown
## 境界線
✅ 常に実行：既存テストの実行、リントの自動修正、コメント追加
⚠️ 事前確認：パッケージ追加・削除、スキーマ変更、環境変数の追加
🚫 禁止：git push -f、.env ファイルの編集、本番データベースへの直接アクセス
```

## コンテキストエンジニアリングとの融合

SDDの次のレイヤーとして、**コンテキストエンジニアリング**という概念が登場している。

Anthropicが提唱したこの概念の定義は明快だ。

> 「最小限の高信号トークンセットで、望ましい結果の可能性を最大化すること」

プロンプトエンジニアリングが「人間とLLMの単発対話を最適化する」技術だとすれば、コンテキストエンジニアリングは「**エージェントと環境全体の情報フローを設計する**」技術だ。

### コンテキスト腐敗の問題

Anthropicはコンテキストウィンドウの拡大に伴う「**コンテキスト腐敗**」現象を警告している。コンテキストウィンドウが大きくなっても、LLMは長いコンテキストの後半では情報を正確に想起できなくなるリスクがある。

これはAIに「仕様書全部を読んでください」と渡すだけでは不十分である理由だ。**必要な情報を必要なタイミングで提供する**設計が重要になる。

### 3つのコンテキスト管理技術

```
Just-in-Time リトリーバル
  → 「この実装にはAPI仕様のみ提供する」
  → 全仕様書を一度に渡さず、タスクに必要な部分だけを抽出

会話履歴のコンパクション
  → 長い会話を要約して圧縮
  → 「前の会話で決まったこと」をサマリとして引き継ぐ

サブエージェントアーキテクチャ
  → 大きなタスクを専門エージェントに分割
  → 各エージェントが扱うコンテキストを最小化
```

## GitHub Spec Kit：実践的なSDD実装

GitHubはSDDを実践するための**Spec Kitオープンソースツールキット**を提供している。

Spec Kitは3種類の仕様書ファイルを中心に構成されている。

### requirements.md

プロジェクトの機能要件と非機能要件を定義する。

```markdown
# 要件定義

## 機能要件
- ユーザーはメールアドレスとパスワードで登録できる
- 認証済みユーザーは記事を CRUD できる
- 記事には Markdown を使える

## 非機能要件
- ページロード: LCP 2.5秒以内
- 可用性: 99.9%
- APIレスポンス: p99で500ms以内
```

### design.md

技術的な設計判断を記録する。

```markdown
# 技術設計

## アーキテクチャ決定
- フロントエンド: Astro（SSG）+ TypeScript
  - 理由: 静的サイト生成によるパフォーマンス優先
- データベース: PostgreSQL（Cloudflare D1）
  - 理由: エッジ環境での低レイテンシ
- 認証: JWTトークン（有効期限24時間）
```

### tasks.md

実装タスクを小さなレビュー可能な単位に分割する。

```markdown
# タスク一覧

## Wave 1（並列実行可）
- [ ] T01: データベーススキーマ定義（担当: DB エージェント）
- [ ] T02: 認証APIエンドポイント設計（担当: API エージェント）

## Wave 2（Wave 1完了後）
- [ ] T03: ユーザー登録実装（T01 依存）
- [ ] T04: JWTミドルウェア実装（T02 依存）
```

これらのファイルが「人間とAI双方が参照できる契約」として機能する。

## 実践：Claude Codeでのお勧めSSDワークフロー

Claude Codeを例に、実際のSDDワークフローを示す。

### ステップ1：CLAUDE.md（プロジェクト憲法）の作成

プロジェクトルートに`CLAUDE.md`を作成し、6つの必須要素を記述する。これが全セッションを通じてClaudeが参照する「プロジェクト憲法」となる。

```markdown
# CLAUDE.md

## コマンド
...（6つの必須要素を記述）

## このプロジェクトで絶対にやってはいけないこと
- git push --force
- 本番環境変数の直接変更
- テストをスキップしたコミット
```

### ステップ2：実装前にスペック作成セッションを設ける

コードを書く前に、AIと一緒に仕様書を作るセッションを設ける。

```
人間: 「認証機能を実装したい。まず要件と設計の仕様書を作ろう」

Claude: 「わかりました。以下を確認させてください：
  - 対象ユーザーは誰ですか？
  - ソーシャルログインは必要ですか？
  - パスワードリセットの仕組みは？
  - セッション管理はどうしますか？」
```

この対話を通じて`requirements.md`と`design.md`が完成する。

### ステップ3：タスクを小さく切り、一度に一つ実装

Addy OsmaniのLLMコーディングワークフローでも強調されているように、「**大きなタスクを小さなチケットに分割して1つずつ処理する**」ことがスコープ管理の鍵だ。

```
❌ 悪い例: 「認証機能全体を実装してください」
✅ 良い例: 「tasks.md のT03（ユーザー登録エンドポイント）のみ実装してください。
           requirements.md の §2.1 と design.md の §3 を参照してください」
```

### ステップ4：コードではなく仕様書をレビューする

SDDで最も重要な文化的変化がこれだ。**コードレビューの前に仕様書レビューを行う**。

仕様書が間違っていれば、どれだけ優秀なコードも間違っている。逆に仕様書が正確であれば、AIが生成したコードの正しさを仕様書で検証できる。

## なぜ今なのか：AI時代特有の必然性

仕様書の重要性は「AI時代以前」も変わらない普遍的な真理だ。しかしAI駆動開発において、その重要性は桁違いに増している。

### 増幅器としてのAI

AIはコンテキストを増幅する。良いコンテキスト（仕様書）が与えられれば、AIは信じられないほど生産的になる。逆に悪いコンテキスト（曖昧なプロンプト）が与えられれば、AIはその悪さを増幅した実装を返す。

### エラーコストの非対称性

手動でコードを書く場合、開発者は書きながら矛盾に気づき修正できる。しかしAIに丸ごと実装させた場合、仕様の矛盾は数百行のコードに展開された後で発見される。修正コストが桁違いに大きくなる。

### マルチエージェント化の加速

2026年現在、開発においてマルチエージェントの活用が急速に広まっている。McKinseyが「4万人の人間社員と2万体のAIエージェントが協働する」と発表したように、AIエージェントが組織の一員として機能し始めている。

マルチエージェント環境では、共有された仕様書なしに複数のエージェントが協調することは不可能だ。仕様書は**エージェント間の共通言語**として機能する。

## 仕様書の品質を上げる実践的なヒント

### 外部動作を定義する

Thoughtworksが推奨するのは、「実装の内部ではなく外部から観察できる動作」を仕様書に書くことだ。

```markdown
# 良い仕様書の書き方

## ユーザー登録 API（POST /api/users）

### 事前条件
- メールアドレスが未登録であること
- パスワードが8文字以上であること

### 事後条件
- ユーザーがデータベースに保存されている
- 確認メールが送信されている
- レスポンスにJWTが含まれている

### 不変条件
- パスワードはハッシュ化して保存する（平文不可）
- メールアドレスは小文字に正規化する
```

### 「禁止事項」を明示する

antirez（Redis作者）が指摘しているように、「良く見えるが悪い解決策についてのヒント」を仕様書に含めることが重要だ。

```markdown
## 禁止パターン
- ❌ グローバル変数の使用（代わりに依存性注入を使う）
- ❌ setTimeout での非同期制御（Promiseを使う）
- ❌ any 型のキャスト（型推論かunionを使う）
- ❌ データベース直接アクセス（必ずリポジトリ層を経由する）
```

### 仕様書を生きたドキュメントとして維持する

仕様書は一度書いたら終わりではない。コードと同期して更新し続けることが重要だ。GitHubは仕様書を「生きた実行可能なアーティファクト」と表現している。

プルリクエストのレビューで「このコード変更は仕様書のどの部分に対応するか？」を確認する習慣をつけることで、コードと仕様書の乖離を防げる。

## まとめ：「計画の1時間が後の10時間を節約する」

ThoughtworksのSDD解説で引用されている原則が、AI駆動開発でも完全に当てはまる。

> 「計画に1時間投資することで、後の10時間の手戻りを節約できる」

AIは道具として強力だが、それを操る人間の設計力・仕様策定力が問われる時代が来た。Vibe Codingの気軽さとSDDの厳密さを使い分けながら、AIと協働するための「仕様書という共通言語」を習得することが、2026年の開発者にとって不可欠なスキルになっている。

コードはAIが書ける。しかし「何を作るべきか」を定義するのは、依然として人間の仕事だ。
